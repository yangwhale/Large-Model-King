# 合阵修炼：分布式并行的六维体系

> 一头神兽的兽卵动辄几百亿甚至上万亿参数，一块灵石（GPU）的丹田（显存）装不下。怎么办？把修炼任务拆开，分给一群灵石同时干。这就是"合阵修炼"——分布式并行训练。

---

## 为什么需要合阵？

一个简单的算术：
- GPT-3 有 175B 参数，FP16 存储需要 350GB 显存
- 一块 A100 只有 80GB 显存
- 就算不算优化器状态和激活值，至少需要 5 块 A100 才能放下模型
- 加上优化器状态（Adam 需要额外 2 倍参数量的显存），至少需要 15 块
- 再加上激活值（中间计算结果），实际上需要 **几十到几百块**

DeepSeek V3 有 671B 参数，训练用了 2048 块 H800。这就像 2048 个修炼者同时围坐一圈，合力孵化一头超级神兽。

问题是：**怎么分工？**

---

## 第一维：数据并行 — DP（Data Parallelism）

**大白话**：每个灵石都装一份完整的模型（神兽的完整拓印）。每个灵石喂不同的灵食（不同的训练数据）。算完之后把大家的心得（梯度）合在一起。

**比喻**：想象你要训练一头幼兽学会 1000 本书。你复制了 8 头一模一样的幼兽，每头读 125 本书。读完之后 8 头幼兽开会分享心得，然后同步更新。

**优势**：简单，通信量不大（只需要同步梯度）
**劣势**：每块 GPU 都要存一份完整模型，模型太大就放不下

**代表框架**：PyTorch DDP（DistributedDataParallel）

---

## 第一维·改：全分片数据并行 — FSDP（Fully Sharded Data Parallelism）

**大白话**：DP 的改良版。模型不再完整复制到每块 GPU——把参数、梯度、优化器状态全部切碎，分散到不同 GPU 上。需要算某一层的时候，临时把这层的碎片从各处聚合起来，算完再打散。

**比喻**：8 头幼兽不再各自背一套完整功法了。每头只背功法的 1/8，需要用哪招的时候现场拼凑。省了 8 倍记忆，但每次出招都要跟同伴沟通一下。

**关键操作**：All-Gather（聚合碎片）+ Reduce-Scatter（分散梯度）
**优势**：显存省到极致（近乎线性减少）
**劣势**：通信量大。但通信可以跟计算重叠（下一层的 All-Gather 跟当前层的计算同时进行）

**代表框架**：PyTorch FSDP, DeepSpeed ZeRO

---

## 第二维：张量并行 — TP（Tensor Parallelism）

**大白话**：把模型的每一层**横着切开**，分给多块 GPU。比如一个矩阵乘法 Y = XW，把 W 按列切成 W1 和 W2，两块 GPU 各算一半，最后拼起来。

**比喻**：一个修炼者要搬一块千斤巨石（大矩阵运算）。一个人搬不动。TP 是把巨石劈成两半，两个人各搬一半，搬到目的地再拼起来。

**通信**：每层都需要一次 All-Reduce（结果汇总）。这在延迟敏感的推理阶段很关键——TP 要求 GPU 之间有极高带宽（NVLink 级别）。

**优势**：降低单层的延迟（因为并行计算）
**劣势**：每层都要通信，通信在关键路径上（不能跟计算重叠）

**代表框架**：Megatron-LM

---

## 第三维：流水线并行 — PP（Pipeline Parallelism）

**大白话**：把模型**纵着切开**——前 10 层放在 GPU 0，中间 10 层放在 GPU 1，最后 10 层放在 GPU 2。数据像流水线一样依次通过每段。

**比喻**：把一条长长的流水线切成三段，三个工人各负责一段。工人 1 做完第一道工序，把半成品传给工人 2，同时开始做下一个产品的第一道工序。

**关键问题**：流水线气泡（Pipeline Bubble）。当工人 1 在做第一个产品时，工人 2 和 3 在干等。GPipe 和 PipeDream 通过微批次（micro-batch）技术来减少气泡。

**优势**：通信量极小（只传层间激活值），适合跨节点
**劣势**：气泡浪费算力。负载均衡困难（不同层计算量不同）

---

## 第四维：专家并行 — EP（Expert Parallelism）

**大白话**：专门为 MoE（混合专家）架构设计的。把不同的专家模块放在不同的 GPU 上。每个 token 根据路由器的指令，被发送到它该去的专家 GPU 上处理。

**比喻**：一个医院有 256 个科室（专家）。病人（token）来了先去分诊台（路由器），分诊台说"你这是心脏问题，去心内科（专家 37）"。病人就被送到心内科的诊室（对应的 GPU）去处理。

**关键通信**：All-to-All。每块 GPU 要把自己的 token 发给正确的专家所在的 GPU，同时接收其他 GPU 发来的属于自己专家的 token。这是 MoE 训练中通信最密集的操作。

**优势**：MoE 的必需品。让万亿参数模型成为可能。
**劣势**：All-to-All 通信是性能瓶颈。专家负载不均衡也是老大难。

**代表工具**：DeepEP（DeepSeek 开源的专家并行通信库）

---

## 第五维：序列并行 — SP（Sequence Parallelism）

**大白话**：把输入序列（一句话的 token 序列）切开，分给多块 GPU。每块 GPU 处理序列的一部分。

**比喻**：一篇万字长文，4 个人各读 2500 字。但问题是 Attention 机制需要每个字都"看到"其他所有字——怎么办？用 Ring-Attention 让 4 个人轮流传阅彼此的笔记。

**两种实现**：
- **Megatron-SP**：跟 TP 配合，在 LayerNorm 和 Dropout 操作上沿序列维度切分
- **Ring-Attention / DeepSpeed-Ulysses**：把 KV 分块，GPU 之间环形传递

**优势**：处理超长序列（百万 token 上下文）的必需品
**劣势**：通信量大，实现复杂

---

## 第六维：上下文并行 — CP（Context Parallelism）

**大白话**：SP 的升级版。专门为超长上下文（1M+ token）设计。在 Attention 计算中把上下文切分到多块 GPU。

**比喻**：SP 是把一本书的页面分给几个人读。CP 更进一步——在"回忆全文"（Attention）的时候，每个人只负责回忆自己那部分的关键信息，然后大家拼起来。

**代表**：DeepSeek V4 的 CSA（Chunked Sparse Attention）+ HCA（Hierarchical Context Attention）就是 CP 的极致运用。

---

## 混合并行：从 3D 到 6D

实际训练中不会只用一种并行。大模型训练通常把多种并行组合起来：

| 维度 | 简称 | 切什么 | 通信类型 | 适用范围 |
|------|------|--------|---------|---------|
| 数据并行 | DP/FSDP | 数据集 | All-Reduce/All-Gather | 同节点内或跨节点 |
| 张量并行 | TP | 层内矩阵 | All-Reduce | 同节点内（需高带宽） |
| 流水线并行 | PP | 层间切割 | 点对点传激活 | 跨节点（低带宽可用） |
| 专家并行 | EP | MoE 专家 | All-to-All | 跨节点 |
| 序列并行 | SP | 序列长度 | Ring/All-Gather | 同节点内 |
| 上下文并行 | CP | 上下文块 | Ring/P2P | 跨节点 |

**3D 并行** = DP + TP + PP（经典组合，Megatron-LM 标配）
**4D 并行** = 3D + EP（MoE 模型必备）
**5D 并行** = 4D + SP 或 CP（超长序列必备）
**6D 并行** = DP + TP + PP + EP + SP + CP（全套拉满，DeepSeek V4 级别）

**HSDP**（Hybrid Sharded Data Parallelism）：在节点内用 FSDP（全分片），跨节点用 DP（数据并行）。结合两者优势——节点内省显存，跨节点省通信。

## 修仙比喻总汇

| 并行方式 | 修仙术语 | 一句话比喻 |
|---------|---------|-----------|
| DP | 分身齐修 | 复制多头兽，各喂不同灵食，定期同步心得 |
| FSDP | 分藏共修 | 功法切碎分人背，用时现场拼，省记忆 |
| TP | 分体并修 | 巨石劈两半，各搬一半，到了再拼 |
| PP | 流水育兽 | 流水线三段，做完传给下一个 |
| EP | 分专精修 | 医院分诊，病人去对应科室 |
| SP | 断句分读 | 长文切段，各读一段，轮流传笔记 |
| CP | 分忆合参 | 回忆全文时各忆一段，拼起来 |
| HSDP | 内聚外散 | 同桌全分享，跨桌只传结论 |
| 3D 并行 | 三维合阵 | 数据+层内+层间三管齐下 |
| 6D 并行 | 六维天阵 | 六种并行全拉满，几千灵石协同 |
