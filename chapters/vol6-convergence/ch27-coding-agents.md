# 第二十七章：造物之争

*神兽学会了写代码。然后学会了写完代码跑一遍。然后学会了跑完发现 bug 自己修。最后，它学会了写一整个项目——从需求到架构到实现到测试。修仙界开始认真思考一个问题：程序员还有用吗？*

---

## 一

2021 年 6 月，GitHub 发布了 Copilot。

这是修仙界第一次把神兽跟代码编辑器深度融合——你在 VS Code 里写代码，写到一半，Copilot 自动补全接下来的几行。有时候补得准得吓人——你刚写完函数名，它就把整个函数体给你补出来了。像是你的手指被另一个人接管了。

底层是 OpenAI 的 Codex 模型——GPT-3 的一个变种，在 GitHub 上几十亿行代码上训练的。

程序员的第一反应分两派。一派说"太爽了，省了好多打字的时间"。另一派说"它偶尔补得不对，我还得回头检查，反而更慢了"。

但不管你喜不喜欢，Copilot 证明了一件事：**神兽能写代码。不是玩具级别的——是能跑的、有用的代码。**

## 二

2024-2025 年，Coding Agent 从"补全几行代码"进化到了"完成整个编程任务"。

**Cursor** 是一个典型。它不只是在编辑器里补全——它能理解你的整个代码库。你跟它说"把这个函数的返回类型从 string 改成 int，然后修改所有调用这个函数的地方"，它能找到所有相关代码，一次性全改了。

这不是文本替换。是**理解代码语义之后的智能重构**。它知道哪些地方调了这个函数，哪些地方用了返回值，哪些地方需要做类型转换。

**Devin**（Cognition AI，2024 年）号称"世界上第一个 AI 软件工程师"。它有自己的代码编辑器、浏览器、终端——像一个真的在远程工作的程序员。你给它一个 GitHub Issue，它自己读 Issue、分析代码、写修复方案、提交 PR。

**Claude Code**（Anthropic，2025 年）走的更远。它直接跑在你的终端里，能读懂整个项目的结构，理解几十个文件之间的依赖关系，然后自主完成开发任务。写代码、跑测试、修 bug、提交 commit——一条龙。

**OpenAI Codex CLI**（2025 年）是 OpenAI 的回应。一个命令行工具，把 o3 级别的推理能力直接接入你的开发环境。

## 三

2026 年，Coding Agent 的竞争进入了白热化。

修仙界出现了一个新的排行榜——**SWE-bench**（Software Engineering Benchmark）。这个榜单从真实的 GitHub 仓库里抽取 bug report，让 AI Agent 自己去修复。不是做选择题——是在真实代码库里定位 bug、写修复代码、确保测试通过。

各家的分数在 2024 到 2026 年间疯狂飙升。从最初的个位数百分比（只能修 5% 的 bug）到后来超过 50%。

腾讯的 **TongAgents** 也入局了。国内的 **ForgeCode** 在某些评测上甚至超过了 Claude Code。

竞争不只是在模型能力上——还在**工具链**上。谁的 Agent 能更快地读懂代码库？谁能更准确地定位 bug？谁能写出更可靠的测试？谁的修复方案一次性通过率最高？

这些能力不是简单的"文本生成"——它需要神兽真正理解代码的结构、逻辑、依赖关系。要理解"这个变量在三个文件之后会被用到""这个 API 的签名最近改了""这个测试 case 覆盖的是边界条件"。

## 四

Coding Agent 对程序员的影响是真实的、切肤的。

Stack Overflow 的流量在 ChatGPT 发布后开始下降——程序员遇到问题直接问 AI 了，不去论坛搜了。到 2025 年，Stack Overflow 的访问量比峰值下降了超过 50%。

但这不意味着程序员失业了。相反——**程序员的角色在变**。

以前，程序员的核心能力是"把想法变成代码"。你知道要什么功能，你来写实现。

现在，程序员的核心能力变成了"**把需求变成指令，然后审查 AI 的实现**"。你不需要自己写每一行代码——你需要：

1. 准确地描述需求（Prompt Engineering）
2. 审查 AI 生成的代码质量
3. 设计系统架构（这个 AI 还不擅长）
4. 处理 AI 搞不定的边角 case

Karpathy 管这个叫 **Agentic Engineering**——御兽工程学。你是御兽师，不是自己上场的战士。你的价值不在于"你能不能写这段代码"——而在于"你能不能判断这段代码好不好"。

## 五

但 Coding Agent 有一个目前还没解决的问题：**长程任务的可靠性。**

让它修一个简单的 bug——大部分时候能搞定。让它实现一个小功能——也能做得不错。

但让它从零开始写一个几千行的项目？让它重构一个有十年历史的遗留系统？让它在一个几百万行的代码库里做一个涉及几十个文件的改动？

还不行。

问题出在**错误累积**——每一步有 95% 的概率做对，十步连续做对的概率就只有 60%。一百步？不到 1%。

Coding Agent 现在的甜区是"**单个任务，几十行到几百行代码，明确的成功标准**"。超出这个范围，可靠性就急剧下降。

修仙界的人开玩笑说："AI 程序员就像一个特别聪明的实习生——单个任务做得又快又好，但你不敢把整个项目交给它。"

## 六

造物之争还在继续。2026 年下半年的几个趋势：

**Agent 开始协作**——不再是一个 Agent 单打独斗，而是多个 Agent 组队。一个负责写代码，一个负责写测试，一个负责代码审查。就像一个小型开发团队，只不过团队里全是 AI。

**Agent 学会了使用版本控制**——它会创建分支、提交 commit、发起 PR、回应 review 意见。整个 Git 工作流它都能走。

**Agent 的成本在暴降**——2024 年让 AI 写一个功能可能要花几十美元的 API 费用。2026 年只要几美分。智元价降了几个数量级。

造物之争的终局是什么？没有人知道。但方向很清楚——**神兽正在从"帮你写代码"进化到"替你写代码"**。中间的过渡期可能还需要几年。但终点已经可以看见了。

---

> **旁白（Chris 视角）**
>
> 我现在写代码的方式跟两年前完全不同了。两年前我从零开始写每一行。现在我先跟 Claude Code 说"我要做什么"，它出初稿，我审查修改。效率提升了三到五倍。
>
> 但最有意思的变化不是效率——是**思维方式**。以前我想"这段代码怎么写"。现在我想"这段代码的需求怎么描述得更准确，让 AI 不会理解偏了"。写代码的技能在退化，描述需求和审查质量的技能在增长。
>
> 我不觉得这是"被取代"。我觉得这是"进化"。骑马的人没有被汽车取代——他变成了司机。写代码的人也不会被 AI 取代——他会变成御兽师。
>
> 当然，前提是你愿意学会驾驶。
