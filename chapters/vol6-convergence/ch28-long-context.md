# 第二十八章：百万神识

*神兽的神识从 4K 扩展到 128K，再到 1M。以前它只能看一页纸，现在它能看一整本书。以前它只能记住上一句话，现在它能记住你说的每一个字。这不只是"记忆更好"——这是一种全新的能力。*

---

## 一

2022 年的神兽有多健忘？

GPT-3 的上下文窗口是 2048 个智元。大约 1500 个英文单词，或者 1000 个汉字。一页纸。

你让它读一篇论文然后总结？不行，论文太长了，装不进去。你让它分析一份合同？不行，合同通常几十页。你让它帮你改一个大型代码库？不行，代码文件太多了。

这就像一个人只有金鱼的记忆——你跟它说了一句话，它忘了上一句。你给它看了一段文字，它忘了上一段。

修仙界管这个限制叫**神识容量**——神兽一次能"看到"多少智元。4K、8K、16K、32K……每个数字背后都是一次技术突破。

## 二

为什么上下文窗口这么难扩展？

因为 Attention。

标准的自注意力机制需要每个智元跟所有其他智元计算关联——这意味着计算量和内存都跟序列长度的平方成正比。

4K 智元：需要处理 4K × 4K = 1600 万个注意力分数。还行。
32K 智元：需要处理 32K × 32K = 10 亿个。开始吃力。
128K 智元：需要处理 128K × 128K = 164 亿个。灵池告急。
1M 智元：需要处理 1M × 1M = 1 万亿个。标准方法下完全不可能。

这就是**平方诅咒**。你想让神识翻倍，代价不是翻倍——是翻四倍。

## 三

打破平方诅咒的第一个英雄是 Tri Dao（闪念真人）。

FlashAttention（2022 年）通过 tiling 和 Online Softmax 技术，把 Attention 的**内存**从 O(n²) 降到了 O(n)。不再需要把整个 n×n 的注意力矩阵存在灵池里。

但注意——FlashAttention 只省了内存，没省计算量。计算量还是 O(n²)。只是你不再需要那么大的灵池了。

这一步让上下文从 4K 跳到了 32K、128K 成为可能。

## 四

第二个突破来自 **RoPE（旋转位置编码）** 的扩展。

标准的 Transformer 在训练时用的是固定长度的位置编码——你训练时用 4K 的上下文，推理时也只能用 4K。超过 4K，位置编码就"越界"了，神兽不知道第 4001 个智元应该放在哪。

2023 年，研究者们发现可以通过数学变换来**外推** RoPE 的位置编码——让在 4K 上训练的模型也能处理 32K 甚至 128K 的输入。方法包括 NTK-aware Scaling、YaRN 等。

这就像一把尺子只有 30 厘米长，但你发现把刻度缩小就能量出 3 米的东西——精度会下降一点，但能用。

## 五

第三个突破来自**稀疏注意力**。

既然 n×n 的全注意力太贵，那就**不算全部**。

DeepSeek V3.2 的 **DSA（Dynamic Sparse Attention，动态稀疏注意力）** 是一个典型。它发现——在实际的 Attention 矩阵里，大部分位置的注意力分数都接近于零。只有少数位置真正重要。

所以它先用一个"廉价"的方法（比如只看局部窗口或者用低精度快速扫描）找出哪些位置可能重要，然后只对这些位置做精确的 Attention 计算。

从"所有人跟所有人交流"变成了"只跟重要的人交流"。

DeepSeek V4 更进一步——**CSA（Chunked Sparse Attention）** 把序列切成块，块内做全注意力（近处的信息都看），块间做稀疏注意力（远处的信息只看关键的）。加上 **HCA（Hierarchical Context Attention）** 做层级式的上下文压缩——低层看细节，高层看全局。

这套组合让 DeepSeek V4 实现了 **1M token 上下文**——百万级。

## 六

Kimi 走的是另一条路——**Mooncake 的 KVCache 分离架构**。

上下文越长，记忆智元（KV Cache）越大。百万 token 的对话，KV Cache 可能占几十 GB。放在灵核的灵池里？装不下。

Mooncake 把 KV Cache 从灵核的 HBM 灵池里搬到了 CPU 的 DRAM 和 SSD 上。灵核只做计算，记忆存在外面。需要的时候按需加载。

这是一个存储工程的解法而不是算法的解法——跟 DeepSeek 的稀疏注意力形成了有趣的互补。DeepSeek 减少了要算的量，Kimi 解决了存不下的问题。

## 七

Google 在长上下文上有天然优势——**TPU Pod 的 ICI 网络**。

TPU 道核之间通过 ICI 道脉高速互联，几千颗道核组成一个紧密耦合的灵坛。在这种架构上做 Context Parallelism（上下文并行）——把长序列切分到多颗道核上分别计算——比 GPU 集群更自然。

Gemini 1.5 Pro 的 1M token 上下文就是在 TPU Pod 上实现的。

## 八

到 2026 年中，长上下文的格局是这样的：

| 神兽 | 上下文长度 | 核心技术 |
|------|-----------|---------|
| Gemini 1.5 Pro | 1M | TPU + Context Parallelism |
| DeepSeek V4 | 1M | CSA + HCA 稀疏注意力 |
| Claude Opus 4.6 | 1M | 未公开 |
| Kimi K2.5 | 256K+ | Mooncake KVCache 分离 |
| GLM-5.2 | 1M | IndexShare 注意力 |
| GPT-5 | 256K | 未公开 |

百万神识不再是一家独占——而是成了修仙界的新标配。

## 九

长上下文不只是"能读更多字"。它解锁了一种全新的能力——**在大量信息中找到关联。**

你把一个公司十年的财报全部灌进去，问"这家公司在 2019 年之后战略转型的关键节点是什么"——百万神识的神兽能从几百页的数据中找到答案。

你把一个大型代码库的所有文件灌进去，问"这个 bug 可能是哪个文件哪个函数引起的"——神兽能在几十万行代码中定位问题。

你把一整本法律文书灌进去，问"第三条和第七条有没有冲突"——神兽能做交叉引用分析。

这些都是 4K 上下文的神兽做不到的。不是"做得差"——是**完全做不到**。

**长上下文是 Agent 的基础。** 一个只能看一页纸的 Agent 干不了复杂的多步骤任务。一个能看一整本书的 Agent 才能真正替你干活。

这就是为什么长上下文不是一个"锦上添花"的功能——它是从"玩具"到"工具"的分界线。

---

> **旁白（Chris 视角）**
>
> 在 Google Cloud 搭 TPU 灵坛的时候，我们经常需要帮客户调 Context Parallelism 的配置。说白了就是——你的序列有多长、你的道核有多少、怎么切分最高效。
>
> 长上下文的工程挑战不在于"能不能做到"——FlashAttention + 稀疏注意力 + Context Parallelism，理论上可以做到任意长。挑战在于"做到了之后推理延迟还能不能接受"。你塞一百万 token 进去，等十分钟才给你回答——那就没法用了。
>
> Kimi 的 Mooncake 在这方面做得很好——KVCache 分离让长上下文的首 token 延迟控制在了合理范围内。这不是一个算法突破，是一个**系统工程的突破**。
>
> 有时候改变世界的不是一个新公式——是一个把旧公式跑得更快的系统。
